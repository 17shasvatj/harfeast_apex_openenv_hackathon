# HarFeast — Codebase Briefing

## What HarFeast Is

HarFeast is an **RL training environment** built on the [OpenEnv](https://github.com/meta-pytorch/OpenEnv) framework. It simulates a **management consulting engagement** (modeled after [APEX-Agents](https://arxiv.org/abs/2601.14242) consulting tasks) where an LLM agent must navigate files, spreadsheets, and data tools to answer 14 multi-step analytical questions about a fictional food manufacturing company. The agent's answers are scored against deterministic rubrics, producing reward signals for GRPO training.

**Target**: Mercor partner sub-theme ($10K prize) — "Build an OpenEnv environment that captures a complex, real-world multi-step task and train an agent within it to improve performance on APEX-Agents."

## Architecture Overview

```
┌─────────────────────────────────────────────────┐
│  train_harfeast.py / train_harfeast_colab.py    │
│  (TRL GRPOTrainer + rollout_func)               │
└──────────────┬──────────────────────────────────┘
               │ HTTP/WebSocket (OpenEnv protocol)
               ▼
┌─────────────────────────────────────────────────┐
│  harfeast_env/server/app.py  (FastAPI)          │
│  └── HarFeastEnvironment (OpenEnv Environment)  │
│       └── HarFeastOpenEnv (core logic)          │
│            ├── actions.py (8 action handlers)   │
│            ├── rubric.py (rubric scoring)        │
│            └── schemas.py (data structures)     │
└──────────────┬──────────────────────────────────┘
               │ Reads CSV/TXT from disk
               ▼
┌─────────────────────────────────────────────────┐
│  harfeast_world/ or harfeast_worlds/world_XXXX  │
│  (generated by harfeast_synthetic_world_gen.py) │
│  ├── data/*.csv  (9 datasets)                   │
│  ├── documents/*.txt  (6 documents)             │
│  ├── tasks.json  (14 tasks w/ rubrics)          │
│  └── ground_truth.json                          │
└─────────────────────────────────────────────────┘
```

**Flow**: Generator creates synthetic worlds → Server hosts the environment → Trainer runs GRPO rollouts against the server → Agent improves at consulting tasks.

## File-by-File Briefing

### `harfeast_openenv/` — Core Environment Logic

Pure Python, zero external dependencies. This is the heart of the project.

| File | Purpose |
|------|---------|
| **`environment.py`** | Main `HarFeastOpenEnv` class. `reset()` loads a task (from single world or augmented dataset), `step()` dispatches actions, tracks history, manages context windowing (last 5 steps), enforces 20-step limit. |
| **`actions.py`** | 8 action handlers: `files.list`, `files.read`, `spreadsheet.read_range`, `data.filter`, `data.group_by`, `data.add_columns`, `data.compute`, `submit`. Handles CSV parsing, path traversal security, table formatting, filtered dataset chaining. |
| **`rubric.py`** | Deterministic rubric scorer. Extracts expected values from "States that X is Y" patterns, normalizes numbers/currencies/percentages, checks substring matches. Returns 0–100 score. |
| **`schemas.py`** | `ActionResult` (observation + success + error), `StepResult` (observation + prompt + step_count + done + reward + info), `parse_action()`. |

### `harfeast_env/` — OpenEnv Server + Client

Wraps `harfeast_openenv` into the OpenEnv HTTP/WebSocket protocol for remote access by TRL training.

| File | Purpose |
|------|---------|
| **`models.py`** | Pydantic models `HarFeastAction` (JSON string encoding) and `HarFeastObservation` (text + metadata), extending OpenEnv base classes. |
| **`client.py`** | `HarFeastEnv` — OpenEnv `EnvClient` that serializes/deserializes actions and observations over HTTP. |
| **`server/app.py`** | FastAPI app factory via `create_app()`. Reads `HARFEAST_WORLD_PATH` and `HARFEAST_WORLDS_BASE` env vars. |
| **`server/harfeast_environment.py`** | `HarFeastEnvironment` — OpenEnv `Environment` implementation. Wraps `HarFeastOpenEnv`, converts `StepResult` ↔ `HarFeastObservation`. |

### `harfeast_synthetic_world_generator.py` — World Generator

The most substantial file (~1450 lines). Generates complete synthetic consulting worlds:

- **9 CSV datasets**: employee_survey (3000+ rows), equipment_data (~250), quality_losses, plant_labor, bls_wage_benchmark, attached_wage_data, oee_assumptions, plant_unit_sales, aptean_report_data
- **6 text documents**: scrap_rate_report, 3 expert interviews (sarah_jenkins, david_chen, mike_russo), frito_lay_case_study, aptean_report
- **14 tasks** with prompts, rubrics, and computed ground truth
- **Batch mode**: `--batch 40` generates 40 distinct worlds (560 task instances) with `manifest.json` and `all_tasks.json`
- Parameterized via `WorldConfig` (seed, plant cities, scrap targets, wage scales, employee count, etc.)

## The 14 Tasks

| ID | Name | What the Agent Must Do |
|----|------|----------------------|
| task_01 | High-Priority Digital Training Employees | Filter employees by readiness/comfort/willingness thresholds, report counts + role breakdown |
| task_02 | Adjusted Cost of Instability | Calculate abnormal scrap cost / (actual - target scrap %) per plant |
| task_03 | Predictive Maintenance Scrap Impact | Identify qualifying equipment, apply 15% scrap reduction, report new rates by product family |
| task_04 | Digital Lever Agreement + OEE Projections | Read 3 interviews for consensus, project OEE growth per plant |
| task_05 | Labor Cost Analysis | Total labor cost, efficiency gains (division-specific), union demand increases for 3 plants |
| task_06 | Operational Efficiency Analysis | Average inefficient hours per plant, identify most/least efficient |
| task_07 | Productivity Loss Quantification | Weekly→annual loss per role (×52), total company-wide loss |
| task_08 | High-Priority Equipment Quality Losses | Canned veg with >5% scrap + above-median downtime, total losses |
| task_09 | Labor Variance Analysis | Standard vs actual hours, dollar variance using BLS wages, productivity index |
| task_10 | Updated Productivity Loss | Re-calculate annual loss using client-provided wage data |
| task_11 | Technology Investment Impact | Top 5 from Aptean report, apply cumulative growth to plant sales |
| task_12 | Digital Adoption Willingness | Highest/lowest willingness plants + roles, training cost estimates |

## Flags: Concerning / Uncertain Code

### RED — Likely to cause failures

**1. Broken fallback imports** in `harfeast_env/client.py` (line 17) and `harfeast_env/server/app.py` (lines 19-21).
The `except ImportError` fallback tries `from models import ...` which fails when running as a package. The try/except blocks are identical for `openenv` imports — they only catch failures on the local import but the fallback is equally broken.

**2. `eval()` usage** in `actions.py` (lines 415, 445).
`handle_data_add_columns` uses `eval()` with `{"__builtins__": {}}`. The character whitelist + AST validation provide protection but `eval` with empty builtins is known to be bypassable. `handle_data_compute` also uses `eval()` on regex-validated expressions. Low risk for RL training, but worth noting.

**3. `trl.experimental.openenv` API uncertainty** in `train_harfeast.py` (line 106) and `train_harfeast_colab.py` (line 45).
`generate_rollout_completions` is experimental. Its return format (`prompt_ids`, `completion_ids`, `logprobs`) may not match the current TRL version. **Verify against installed TRL before training.**

**4. `rollout_func` output structure** in `train_harfeast.py` (lines 140-145).
The custom `env_reward` key is passed via `reward_func(**kwargs)` — relies on TRL forwarding extra rollout keys to reward kwargs. This is fragile and may silently fail (rewards default to 0.0).

### YELLOW — Subtle bugs possible

**5. Rubric scorer is regex/substring-based** (`rubric.py`).
Can false-positive (number meant for one criterion matching another) or false-negative (equivalent representations the normalizer doesn't cover). `REWARD_FUNCTIONS.md` acknowledges this.

**6. `HarFeastObservation` doesn't declare `done`, `reward`, or `metadata` fields** (`models.py`).
These are passed in `harfeast_environment.py` lines 87-98, presumably inherited from `Observation` base. If the parent doesn't accept them, Pydantic validation will fail.

**7. `_task` can be `None`** — `environment.py` line 292 accesses `self._task['task_name']` in `_build_context_summary()`. If `step()` is called before `reset()`, this crashes instead of returning a graceful error.

**8. No root `.gitignore`** — `harfeast_worlds/` (40 worlds, hundreds of files) is tracked in git, bloating the repo.

### MINOR

**9.** `run_openenv_client.py` name suggests HTTP client testing but uses in-process `HarFeastOpenEnv` directly.

**10.** Hardcoded constants: `MAX_TABLE_ROWS=20`, `MAX_DOCUMENT_CHARS=2000` (actions.py), `CONTEXT_WINDOW_STEPS=5`, `MAX_STEPS=20` (environment.py), `$8/hour` training cost (world generator).

**11.** `SUPPORTS_CONCURRENT_SESSIONS = False` in `harfeast_environment.py` — only one episode at a time (fine for training, blocks parallel eval).

## Quick Start

```bash
# Generate a world
python harfeast_synthetic_world_generator.py --seed 42 --output-dir ./harfeast_world

# Generate augmented dataset (40 worlds, 560 tasks)
python harfeast_synthetic_world_generator.py --batch 40 --output-dir ./harfeast_worlds

# Test locally (demo mode)
python run_environment.py --task task_14

# Test locally (interactive mode)
python run_environment.py --task task_14 --interactive

# Start env server
uvicorn harfeast_env.server.app:app --host 0.0.0.0 --port 8000

# Train with GRPO (local server)
python train_harfeast.py --env-url http://localhost:8000 --model Qwen/Qwen2.5-0.5B-Instruct

# Train with GRPO (HF Space)
python train_harfeast.py --env-url https://openenv-community-harfeast-env.hf.space

# Train with augmented dataset
python train_harfeast.py --env-url http://localhost:8000 --worlds-base ./harfeast_worlds
```
